# 评测工作台前端重构设计

## 1. 文档定位

本文档定义 `cockpit-agent-evaluation-framework` 的新一代前端产品设计。

本次设计不再以现有 5 个页面为基础做增量修补，也不以“重画旧看板样式”为目标，而是将整套 Web 前端重新定义为一套面向测试与算法同学的 **评测工作台**。

本设计关注三个问题：

- 如何让用户从测试集准备开始，一路完成 run 发起、执行观察、结果分析和回归处理
- 如何把 `suite / case / run / result / risk` 这几个核心对象组织成稳定的前端系统
- 如何在保留治理与展示能力的同时，补齐高频操作入口与任务闭环

本设计与以下内容保持一致：

- 平台整体定位为独立 Agent 评测平台
- 本轮优先服务测试与算法同学，而不是平台维护者
- 单主 target 优先，例如 `cockpit_agents`
- `suite / case` 编辑以结构化表单为主，JSON 高级编辑为辅
- 一旦资产被 run 使用，原版本冻结，只能复制出新版本继续修改

补充设计系统基线已生成于：

- `design-system/cockpit-agent-evaluation/MASTER.md`

## 2. 背景与问题

当前前端已经具备若干展示能力，但产品结构存在根本性问题：

- 前端更偏治理与展示层，而不是评测工作台
- 缺少测试集准备与运行创建入口
- 用户需要依赖 `run_id`、`case_id` 手工查询，不适合日常使用
- `suite / case` 的创建和编辑没有形成可用的前端工作流
- 结果、风险、资产之间缺乏稳定跳转关系

现有前端的 5 个页面：

- `目标总览`
- `运行中心`
- `用例回看`
- `趋势`
- `回归`

这些页面代表的业务能力仍然有价值，但它们不应继续作为新前端的信息架构基础。本次设计将它们视为能力线索，而不是需要保留的页面边界、导航结构或视觉风格。

## 3. 设计目标与非目标

### 3.1 设计目标

本次重构需要实现以下目标：

- 让前端成为完整的评测工作台，而不是割裂的展示面板
- 围绕测试与算法同学的主任务链路组织页面和导航
- 支持手工维护 `suite / case`
- 支持从 `cockpit-benchmark` 导入测试包
- 支持 run 创建、run 列表、run 详情、复跑
- 支持结果复盘、case 历史回看、趋势、回归和告警
- 保证评测资产与历史 run 的可追溯性
- 在数据密度较高的前提下保持清晰、稳定、可访问

### 3.2 非目标

本次设计不直接追求以下内容：

- 首版复杂多 target 管理体系
- 以平台维护者为中心的重型后台
- 直接把 runner 领取任务和结果回传做成人工操作页面
- 沿用现有 5 个页面的结构与路由
- 把 JSON 编辑器作为主要编辑入口

## 4. 核心用户与主任务

### 4.1 核心用户

- 主要用户：测试同学、算法同学
- 次要用户：平台维护者

本轮设计优先满足主要用户的高频日常操作，次要用户能力应下沉到低频设置区，不应主导导航结构。

### 4.2 主任务

主要用户最核心的工作链路如下：

1. 准备测试集
2. 手工创建或编辑 `suite`
3. 手工创建或编辑 `case`
4. 从 `cockpit-benchmark` 导入 package
5. 选择环境和测试范围创建 run
6. 查看 run 执行状态与失败项
7. 查看 run 结果、case 历史和归一化结果
8. 查看趋势、回归与告警
9. 对失败或异常 run 发起复跑

前端设计应围绕这条链路构建，而不是围绕底层后端模块构建。

## 5. 整体产品方向

### 5.1 产品定义

新前端定义为：

**Evaluation Workbench**

它是一套以任务闭环为中心的评测操作系统，核心使命是帮助用户完成“准备测试集 -> 创建运行 -> 观察执行 -> 分析结果 -> 处理风险”的完整工作流。

### 5.2 架构方向

本次采用 **工作台优先 + 轻资产中心** 的产品结构：

- `工作台` 作为全局入口，承载快捷动作与当前最重要问题
- `测试集` 作为资产工作区，承载 `suite / case` 的浏览、创建、编辑和导入
- `运行` 作为执行控制区，承载 run 的创建、列表、详情和复跑
- `结果` 作为复盘与证据区，承载 run report、case 历史和结果查看
- `风险` 作为治理区，承载趋势、回归和告警
- `设置` 作为低频配置区，承载 environment、target 基础信息和规则查看

### 5.3 旧页面的处理方式

旧的 5 个页面不保留为固定导航，而是被吸收到新的能力体系中：

- `目标总览` 的能力被吸收进 `工作台`
- `运行中心` 的能力被吸收进 `运行`
- `用例回看` 的能力被吸收进 `结果`
- `趋势` 的能力被吸收进 `风险`
- `回归` 的能力被吸收进 `风险`

本次重构允许：

- 拆分旧页面
- 合并旧页面
- 重命名旧页面
- 改写旧页面布局
- 调整旧页面的入口层级

## 6. 信息架构

### 6.1 一级模块

新的一级导航为：

- `工作台`
- `测试集`
- `运行`
- `结果`
- `风险`
- `设置`

### 6.2 一级模块职责

#### 工作台

负责承载日常工作的起点，而不是做成一个信息堆砌的大盘。

核心职责：

- 告知用户当前最重要的状态
- 提供高频快捷操作
- 将用户送往正确的工作流

典型内容：

- 最近运行
- 最近导入
- 待处理回归与告警
- 快捷动作：导入测试包、新建 Suite、新建 Run、复跑最近失败 Run

#### 测试集

负责评测资产的工作流。

核心职责：

- 浏览 `suite`
- 查看 `suite` 与 `case` 关系
- 手工创建与编辑 `suite / case`
- 导入 benchmark package

#### 运行

负责执行控制。

核心职责：

- 创建 run
- 浏览 run 列表
- 查看 run 详情
- 发起 rerun

#### 结果

负责复盘与证据查看。

核心职责：

- 浏览 run report
- 查看 case 历史
- 对比结果变化
- 查看 normalized results

#### 风险

负责问题发现与处理入口。

核心职责：

- 趋势分析
- 回归队列
- 告警流
- 进入具体 run 与 case

#### 设置

负责低频配置。

首版建议只承载：

- environment 管理
- target 基础配置查看
- 告警规则查看

## 7. 核心对象模型与状态

### 7.1 核心对象

前端必须围绕 5 个核心对象组织：

- `Suite`
- `Case`
- `Run`
- `Result`
- `Risk`

对象关系如下：

- `Suite` 是可复用测试范围，是 run 的选择单位
- `Case` 属于某个 `Suite`
- `Run` 绑定当次选择的 target、environment 与 suite 集合
- `Result` 由 `Run` 执行后生成
- `Risk` 从 `Result` 中提炼而来，包括趋势、回归和告警

### 7.2 资产状态

`Suite / Case` 使用统一资产状态：

- `Draft`
- `Used`
- `Superseded`

含义如下：

- `Draft`：未被任何 run 使用，可原地编辑，可删除
- `Used`：已被 run 使用，只读，不可原地修改
- `Superseded`：已被新版本替代，仅用于历史引用和追溯

### 7.3 执行状态

`Run` 使用统一执行状态：

- `Queued`
- `Running`
- `Succeeded`
- `Failed`

### 7.4 风险状态

`Risk` 使用统一严重程度：

- `Info`
- `Warning`
- `Critical`

### 7.5 状态规则

前端需要遵守以下规则：

- `Draft` 资产可编辑
- `Used` 资产只读
- 对 `Used` 资产继续修改时，应通过“复制新版本”完成
- 历史 run 必须绑定执行时的资产版本，不受后续编辑影响
- 风险项必须从具体 run 和结果上下文中进入，而不是孤立存在

## 8. 视觉系统

### 8.1 风格方向

本次视觉方向采用：

- `Data-Dense Dashboard`
- `Swiss Minimalism`
- 轻量控制台气质

不采用：

- `HUD / Sci-Fi FUI`

原因：

- 数据密集型工作台需要高可读性与高稳定性
- 过度未来感风格在长时间使用中容易带来噪声和可访问性问题
- 本平台需要支持表单编辑、表格浏览、结果比对和风险分析，不适合使用装饰性极强的视觉语言

### 8.2 颜色系统

推荐颜色基线如下：

- Primary：`#1E40AF`
- Secondary：`#3B82F6`
- CTA：`#F59E0B`
- Success：`#22C55E`
- Danger：`#EF4444`
- Dark Shell：`#0F172A`
- Light Workspace：`#F8FAFC`

色彩使用策略：

- 壳层、导航、全局上下文使用深色基底
- 主工作区、表格、表单和详情面板使用浅色高对比背景
- 关键风险、动作和状态使用对比色点亮，而不是全面高饱和

### 8.3 字体系统

推荐字体组合：

- UI 字体：`IBM Plex Sans, Noto Sans SC, sans-serif`
- 数据字体：`JetBrains Mono, monospace`

使用原则：

- 标题、标签、描述、表单使用无衬线字体
- `run_id / case_id / suite_id / 指标 / 状态码` 使用等宽字体
- 中文场景优先保证可读性，而不是强化纯英文开发工具气质

### 8.4 状态颜色语义

- 蓝：主流程、默认状态、主操作
- 绿：通过、成功、健康
- 橙：运行中、待关注、轻风险
- 红：失败、回归、告警

颜色不能成为唯一状态表达，必须同时配合：

- 文本标签
- 图标
- 行内说明

## 9. 布局与交互原则

### 9.1 全局骨架

全站统一采用 4 层骨架：

- 左侧主导航
- 顶部全局上下文栏
- 主内容区
- 右侧详情抽屉或辅助面板

目的：

- 在高密度页面中维持稳定阅读节奏
- 允许列表、详情、历史和证据并行查看
- 降低频繁整页跳转带来的上下文丢失

### 9.2 页面内部结构

每个页面统一遵循以下顺序：

1. 页头：页面目标、当前范围、主操作
2. 筛选层：搜索、状态筛选、时间范围、排序
3. 主工作区：表格、表单、图表、时间线
4. 辅助层：详情抽屉、历史、说明、异常反馈

### 9.3 列表与详情模式

`Suite / Case / Run / Risk` 等对象优先使用：

- 主列表或表格
- 行级点击进入右侧详情抽屉
- 需要更深信息时再进入完整详情页

这样更符合测试与算法同学的大量扫描、快速定位和连续切换行为。

### 9.4 创建与编辑模式

- `Create Run` 使用分步向导
- `Suite / Case` 使用结构化表单编辑
- JSON 高级编辑作为扩展模式，而不是默认主界面

复杂编辑统一遵循：

1. 先填写最少必要信息
2. 再展开高级配置
3. 提交前做预览或确认

### 9.5 跳转原则

前端必须保证以下跳转链路稳定可用：

- 从 `Suite` 可直接进入 `Create Run`
- 从失败 `Case` 可直接进入 `Case 历史`
- 从回归或告警可直接进入对应 `Run Detail`
- 从 `Used` 资产可直接执行“复制为新版本”

设计重点不只是“能看见对象”，而是“用户知道下一步去哪”。

### 9.6 表格原则

高密度表格应统一具备：

- 固定表头
- 行 hover 高亮
- 明确的主操作列
- 行级状态标签
- 批量选择能力预留
- 重要列固定在左侧，例如名称、状态、更新时间

### 9.7 表单原则

表单必须满足：

- 必填字段标识清楚
- 失焦或提交时提供即时校验
- 错误就近展示
- 提交时提供 loading / success / error 反馈
- 复杂字段提供上下文说明，不要求用户理解底层 JSON 才能完成操作

## 10. 模块级页面设计

### 10.1 工作台

#### 工作台首页

职责：

- 作为系统入口
- 汇总当前最重要的运行、导入和风险信息
- 提供快捷动作

页面结构建议：

- 页头：当前 target、今日状态、主操作
- 指标带：最近运行数、失败运行数、待处理风险数
- 快捷动作区：导入测试包、新建 Suite、新建 Run、复跑最近失败 Run
- 最近活动区：最近运行流、最近导入
- 风险摘要区：当前回归、当前告警

### 10.2 测试集

#### Suite 列表

默认入口应为 `Suite` 列表，因为 suite 是 run 的选择单位。

每条 `Suite` 建议展示：

- 名称
- 模式
- case 数量
- 来源类型
- 资产状态
- 最近更新时间

#### Suite 详情

职责：

- 查看 suite 基本信息
- 查看关联 case
- 进入 case 编辑
- 发起 run

#### 新建/编辑 Suite

职责：

- 维护 suite 的基础字段与组织方式
- 控制关联 case 关系
- 明确展示当前状态是 `Draft`、`Used` 还是 `Superseded`

#### Case 列表与详情

职责：

- 作为 `Suite` 下的明细对象浏览入口
- 展示 case 的基本信息、输入概况和最近执行状态

#### 新建/编辑 Case

职责：

- 结构化编辑 `input / expectations / metadata`
- 提供 JSON 高级模式
- 保障普通用户不必直接面对原始 JSON

#### Benchmark 导入页

职责：

- 导入 benchmark package
- 导入前预览 suite、case 数量和来源摘要
- 导入后给出清晰反馈与冲突提示

### 10.3 运行

#### 创建 Run

应做成向导页，而不是散装配置表单。

建议步骤：

1. 选择测试集
2. 选择环境
3. 选择执行方式
4. 提交前预览

提交前必须展示：

- target
- environment
- suite 数量
- case 数量
- 预计任务数
- execution topology

#### Run 列表

职责：

- 成为高频运行入口
- 支持搜索、筛选和排序
- 替代“手输 run_id 查询”的工作方式

#### Run 详情

这是整套系统最关键的页面。

建议组织为 4 个区域：

- 运行摘要
- 套件与用例执行情况
- 结果与回归
- 原始证据与复跑入口

默认优先展示：

- 当前失败项
- 当前异常状态
- 当前风险信号

而不是先展示装饰性图表。

#### Rerun

应作为 run 详情中的主操作之一，允许用户快速重试失败 run 或复验结果。

### 10.4 结果

#### Run Report 浏览页

职责：

- 以复盘视角浏览某次 run 的聚合结果
- 补充 `Run Detail` 中偏操作性的上下文

#### Case 历史页

职责：

- 按 case 查看最近运行历史
- 帮助回答“这个 case 是偶发失败还是稳定退化”

#### 结果对比页

职责：

- 对比不同 run 的结果变化
- 强调可比范围与差异点

#### Normalized Results 查看区

职责：

- 查看结构化归一结果
- 作为 HTTP 结果理解与定位的辅助层

### 10.5 风险

#### 风险中心首页

默认优先级应为：

1. 当前打开的回归
2. 当前打开的告警
3. 趋势概览

#### 趋势页

职责：

- 提供通过率和关键指标的时间趋势
- 作为风险分析的辅助视角，而非唯一入口

#### 回归队列页

职责：

- 展示当前回归信号
- 提供进入 run、case 和复跑的入口

#### 告警流页

职责：

- 展示当前告警事件
- 帮助用户回到具体执行上下文处理问题

### 10.6 设置

首版设置区应保持克制，仅承载低频配置与查看能力：

- Environment 管理
- Target 基础信息查看
- 告警规则查看

## 11. 可访问性与可用性约束

设计必须满足以下约束：

- 使用语义元素优先，不用 `div` 模拟按钮
- 所有交互元素都必须有明确 focus 状态
- 所有图标按钮必须有可访问名称
- 表单控件必须有标签
- 错误信息必须就近展示
- 抽屉、对话框和步骤流必须支持键盘操作
- 异步更新需要有可感知的状态反馈
- 颜色不是唯一状态表达

对导航较重的页面，应考虑：

- `skip to main content`
- 清晰的标题层级
- 稳定的 tab 顺序

## 12. 数据与接口依赖

### 12.1 已有能力

当前后端已具备以下能力，可作为前端设计落地的基础：

- target / environment / suite / case 创建
- benchmark package 导入
- run 创建与 rerun
- run report
- dashboard / trends / regressions / alerts

### 12.2 前端需要但当前缺失的能力

为了支持本设计，后端需要补齐以下接口能力：

- `environment` 列表接口
- `suite` 列表接口
- `case` 列表接口
- `suite` 详情接口
- `case` 详情接口
- `suite` 更新接口
- `case` 更新接口
- `suite` 删除或归档接口
- `case` 删除或归档接口
- run 列表接口
- run task 明细接口
- case 结果明细接口
- artifact 明细接口

这些接口缺口不影响当前设计方向，但会影响实现优先级与页面细节。

## 13. 交付范围

### 13.1 P0

P0 页面与能力：

- 工作台首页
- Suite 列表
- Suite 详情
- Case 编辑器
- Benchmark 导入页
- 创建 Run
- Run 列表
- Run 详情
- 风险中心
- Case 历史

P0 目标是让用户完成完整主链路：

准备测试集 -> 创建运行 -> 观察执行 -> 查看结果 -> 处理风险

### 13.2 P1

P1 可以后置的能力：

- 结果对比页
- 更细的 artifact 浏览页
- 批量操作能力
- 更完整的规则配置页
- 多 target 视图

## 14. 成功标准

本次前端重构完成后，应满足以下标准：

- 测试与算法同学不依赖 `curl` 即可完成完整评测闭环
- 用户不需要记忆 `run_id` 和 `case_id` 才能进入关键对象
- `suite / case / run / result / risk` 的跳转关系稳定可用
- 风险项能够直接带用户回到可处理的问题上下文
- 高密度页面仍然清晰、稳定、可访问
- 历史 run 的结果与资产版本具有可追溯性

## 15. 对设计师的交付要求

基于本设计，后续 UI/UX 设计应至少产出：

- 一份新的信息架构图
- 一份主导航方案
- 一份核心任务流图
- 一组 P0 页面 wireframe
- 一组关键页面高保真稿
- 一份状态矩阵
- 一份对象跳转关系图

设计输出时必须明确回答：

- 用户第一次进入系统会先看到什么
- 用户如何从测试集一路走到 run 创建
- 用户如何从风险项回到具体 run 和 case
- 用户如何识别某个资产是否还能编辑
- 用户如何知道下一步该做什么

## 16. 结论

`cockpit-agent-evaluation-framework` 的前端应被重新定义为一套 **评测工作台**：

- 它不是旧 5 个展示页的视觉翻新
- 它不是以管理员为中心的重型后台
- 它不是单纯 BI 意义上的监控看板

它应是一套服务测试与算法同学日常工作的前端系统，稳定支撑：

- 测试集准备
- 运行发起
- 执行观察
- 结果分析
- 回归与告警处理

这一定义是本次前端系统重构的唯一中心。
